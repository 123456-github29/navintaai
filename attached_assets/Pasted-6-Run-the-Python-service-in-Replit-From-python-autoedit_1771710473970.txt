4) Node endpoint: “Create AutoEdit Job”

You need an endpoint like:
	•	POST /api/videos/:id/autoedit

Request body
	•	clipPaths: string[] (Supabase storage paths, not URLs)
	•	aspectRatio: "9:16" | "1:1" | "16:9"
	•	optional stylePackId

Response
	•	{ jobId } (202 accepted)

What this endpoint does internally
	1.	Verify session JWT
	2.	Verify ownership
	3.	Create a job row: render_jobs with status=queued
	4.	Kick worker/async job runner

⸻

5) Worker step A — Generate signed URLs (Supabase)

For each clip path, create a signed URL:
	•	expiry: 10–30 minutes is enough
	•	never store signed URLs permanently, regenerate on demand

If you have multi-clip videos, you’ll either:
	•	analyze each clip individually (better)
	•	or stitch to a temporary concat mp4 for analysis (easier, but extra work)

Best route for quality: analyze each clip individually, then merge EDL.

⸻

6) Worker step B — Call the Python “AutoEdit” service

Python endpoint
	•	POST /autoedit

Payload (recommended)
	•	fps
	•	clips: [{ url, id }]
	•	goalHints (optional: “fast pacing”, “talking head”, etc.)

Return
	•	EDL JSON (all frames as integers)

This EDL becomes your single “source of truth” for Remotion.

If you render inside Node:

11) Security + anti-abuse (must do)

A) JWT verification

Every “create job” call must:
	•	read Authorization: Bearer <supabase_access_token>
	•	validate via Supabase server client

B) Rate-limit expensive endpoints

At least:
	•	max renders per user per hour
	•	max clip size
	•	max duration

C) Signed URL TTL

10–30 minutes. Never longer.

D) Input validation
	•	clipPaths are strings
	•	limit number of clips (ex: <= 12)
	•	enforce max duration per clip (ex: <= 90s)

E) Background worker only

Never render on the API request thread.
